# 1. Vertex Lighting
**조명 연산을 정점(Vertex) 단계에서 수행**하는게 특징인 라이팅

- 연산의 **결과값을 픽셀(Fragment) 단계로 보간(Interpolate)**하여 전달하는 방식

---

**배경**: 1990년대 하드웨어 제약으로 인해 CPU에서 연산하던 방식에서 유래 (예: PS1 스타일)
   - **장점**: 연산량이 매우 적어 성능 최적화에 유리
   - **단점**: 정점 사이의 조명값이 선형 보간되므로, 메쉬의 밀도가 낮으면 각진 아티팩트(Artifacts)가 발생


---

## Vertex Lighting vs Pixel Lighting

| 구분    | Vertex Lighting         | Pixel Lighting         |
| ----- | ----------------------- | ---------------------- |
| 계산 위치 | Vertex Shader           | Pixel(Fragment) Shader |
| 계산 단위 | 정점                      | 픽셀                     |
| 품질    | 낮음 (보간 의존)              | 높음 (정밀 계산)             |
| 성능    | 연산량이 적어 구형 하드웨어에 적합    | 상대적으로 느림               |
| 아티팩트  | 있음 (edge interpolation) | 거의 없음                  |

## 사전 준비
머티리얼 그래프에서 출력 노드의 셰이딩 모델을 `unlit`으로 변경

- 엔진의 **기본 Lit 셰이딩을 쓰지 않고, 내가 최종 색을 직접 계산해서 Emissive로 내보내겠다!**
- 진짜 언리얼 라이트 계산을 바꾼 것이 아니라, 기본 Lit 경로를 우회해서 직접 색을 그리는 것에 가까움

### unlit
Unlit 셰이딩 모델을 선택하는 순간, 엔진은 **해당 머티리얼에 대해 모든 표준 라이팅 계산을 수행하지 않도록 설계**

- 씬의 모든 오브젝트를 Unlit으로 변경하면 다음과 같은 기능들을 사용할 수 없게 됨

```
- 다이내믹 그림자
- Reflection Captures, Planar Reflections
- 글로벌 일루미네이션 (Lumen 등)
- 서브서피스 스캐터링 (Subsurface Scattering) 등의 고급 셰이딩 모델
```

## vertex Lighting 구현 흐름

```
[Vertex Shader]
    ↓
Ambient 계산
    ↓
Diffuse 계산 (Lambert)
    ↓
합산
    ↓
Custom Interpolator 전달
    ↓
[Pixel Shader]
    ↓
Texture와 곱
    ↓
출력
```

## 로직 코드로 이해하기
### Vertex light
VS(연산) -> 보간(Color) -> PS(출력)

```c++
// [Vertex Shader]
struct VS_OUT {
    float4 pos : SV_POSITION;
    float3 litColor : COLOR; // 연산 결과(색상)를 보간용으로 넘김
};

VS_OUT VS_Main(VS_IN input) {
    VS_OUT output;
    output.pos = mul(input.pos, worldViewProj);

    // 1. 정점 레벨에서 라이팅 계산 (Lambert)
    float3 N = normalize(input.normal);
    float3 L = normalize(lightDir);
    float intensity = saturate(dot(N, L));
    
    // 2. 결과 색상을 확정하여 출력
    output.litColor = ambient + (diffuseColor * intensity); 
    return output;
}

// [Pixel Shader]
float4 PS_Main(VS_OUT input) : SV_TARGET {
    // 하드웨어가 보간해준 색상을 그대로 출력 (추가 연산 없음)
    return float4(input.litColor, 1.0f);
}
```

---

### Pixel light
VS(데이터 전달) -> 보간(Normal) -> PS(연산)

```c++
// [Vertex Shader]
struct VS_OUT {
    float4 pos : SV_POSITION;
    float3 normalWS : TEXCOORD0; // 가공되지 않은 '노멀'을 보간용으로 넘김
};

VS_OUT VS_Main(VS_IN input) {
    VS_OUT output;
    output.pos = mul(input.pos, worldViewProj);
    output.normalWS = mul(input.normal, (float3x3)worldMatrix); // 노멀만 전달
    return output;
}

// [Pixel Shader]
float4 PS_Main(VS_OUT input) : SV_TARGET {
    // 1. 보간된 노멀은 길이가 변했을 수 있으므로 재정규화 필수
    float3 N = normalize(input.normalWS);
    float3 L = normalize(lightDir);
    
    // 2. 픽셀 레벨에서 라이팅 계산 (매 픽셀마다 정밀하게 수행)
    float intensity = saturate(dot(N, L));
    float3 finalColor = ambient + (diffuseColor * intensity);
    
    return float4(finalColor, 1.0f);
}
```

## 사용 노드
### Vertex Interpolator
특정 연산을 **픽셀 셰이더(Pixel Shader)**가 아닌 **버텍스 셰이더(Vertex Shader)**에서 수행하도록 강제

- **Vertex Shader** : 정점 단위 연산이라 연산 횟수 적음
- 고비용 연산(예: 복잡한 수학식, 라이팅 연산)을 버텍스 단계로 옮겨 픽셀 셰이더의 부담을 줄임

---

### SkyAtmosphereLightDirection[0]
Directional Light 중 '하늘 대기(Sky Atmosphere)' 시스템과 연동된 태양의 방향 벡터

- **방향 벡터** : 이 노드는 정규화된(Normalized) 방향 벡터 값을 출력
- **Light Source 설정** : 레벨에 있는 디렉셔널 라이트의 디테일 패널에서 Atmosphere Sun Light 옵션이 켜져 있어야 이 노드가 값을 정상적으로 받아옴

# half-Lambert
Vertex Lighting의 보간 아티팩트를 줄이고, 더 부드러운 명암을 만들기 위해
> 조명의 분포 방식을 의도적으로 바꾸는 스타일라이즈드 라이팅 기법

- Valve가 Half-Life, Half-Life 2, Team Fortress 2 등에서 사용

---

## half-Lambert 핵심 개념
`dot(N, L)`의 **범위를 재매핑**해서 **명암 변화가 더 넓게 퍼지도록** 만드는 방식

```c++
halfLambert = dot(N, L) * 0.5 + 0.5;
```

1. dot(N, L) 계산 → 범위 : [-1 ~ 1]
2. `* 0.5`  → 범위 : [-0.5 ~ 0.5]
3. `+ 0.5 ` → 범위 : [0 ~ 1]

- 원래는 음수였던 뒷면 값도 0 근처의 양수로 올라옴
- 명암 변화가 절반 구면이 아니라 전체 구면에 걸쳐 더 넓게 분포
- 결과적으로 조명이 뒤쪽까지 살짝 감기는 듯한(wrap) 느낌이 생김

---

### 왜 Vertex Lighting에서 특히 유용할까?
Vertex Lighting은 정점에서 계산한 값을 픽셀로 보간 → Lambert처럼 변화가 급한 함수는 정점 수가 적을수록 경계가 깨져 보임

따라서 Half-Lambert는 변화 기울기를 완만하게 만들어:
- 정점 간 값 차이를 줄여 보간 결과를 덜 거칠게 만들고 terminator 부근 아티팩트를 완화

---

### Half-Lambert의 한계
1. **물리 기반 렌더링에는 부적합**
    - 실제 빛처럼 동작하지 않음
    - 빛이 닿지 않는 뒤쪽도 밝아지기 때문
2. **콘트라스트가 줄어듦**
    - 드라마틱한 명암 대비가 필요한 장면에서는 힘이 빠져 보임
3. **리얼리스틱 표현에는 어색할 수 있음**
    - 현실적 재질 표현보다는 NPR, 레트로, 스타일라이즈드 쪽에 잘 맞는다
