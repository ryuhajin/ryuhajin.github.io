## Nanite가 왜 나왔는가
기존 파이프라인은 크게 보면 이런 문제가 있었다

1. **수동 LOD 제작 비용**
    - 자산마다 LOD를 만들고 검수하는 비용이 큼
2. **노멀맵 베이킹 중심 워크플로**
    - 실제 형상을 버리고 텍스처로 속이는 과정이 많음
3. **고폴리 자산 사용의 현실적 한계**
    - 영화급/스캔 자산을 게임에 바로 넣기 어려움
4. **드로우콜/폴리곤/메모리 예산 관리 부담**
    - 얼마나 줄여야 실시간으로 돌아가나를 사람이 계속 검증하고 계산해야 함

> 따라서 Nanite는 고품질 원본 자산을 넣고, 엔진이 보이는 디테일만 잘 골라서 자동으로 처리하자! 라는 방향으로 나온 기술

---

### 기존 렌더링 파이프라인 vs Nanite 파이프라인
- 기존

```
원본 메시
-> 게임용 저폴리 메시 제작
-> 필요하면 노멀맵 베이크
-> 여러 단계 LOD 제작
-> 배치/인스턴싱/HLOD 구성
-> 런타임에 CPU가 오브젝트 단위로 가시성 판단
-> 선택된 LOD를 GPU가 일반 래스터라이즈
```

> 어떤 LOD를 보여줄지 / 어떤 메시를 합칠지 / 어떤 디테일을 노멀맵으로 때울지를 사람이 제작하고 결정

---

- nanite

```
고폴리 원본 메시
-> Nanite 빌드(내부 포맷 생성, 계층 구조 생성, fallback mesh 생성 가능)
-> 런타임에 필요한 클러스터만 디스크/메모리에서 스트리밍
-> GPU가 클러스터 단위로 가시성/크기/오클루전 판정
-> 살아남은 클러스터만 Nanite 경로로 래스터라이즈
-> 머티리얼/섀도우 패스와 연동
```

> 메시 내부의 작은 클러스터 묶음을 엔진이 직접 관리하고 처리한다 = 자동화

---

### Nanite와 기존 메시의 렌더링 파이프라인 차이 정리
1. **입력 자산 개념이 다르다**
    - 기존: 게임용 메시 + LOD 체계
    - Nanite: 원본 메시 -> Nanite 내부 포맷
2. **가시성 제어 단위가 다르다**
    - 기존: 메시/섹션/LOD 단위 비중이 큼
    - Nanite: 클러스터 단위
3. **렌더링 경로의 중심이 다르다**
   - 기존: 선택된 LOD 메시를 일반 래스터 경로로 그림
   - Nanite: 살아남은 클러스터를 Nanite 경로로 그림, 그리고 material 단위의 대규모 처리 방식이 강해짐

> 기존 방식 = 미리 정해 둔 몇 개의 저해상도 버전(LOD) 중 하나를 고르는 방식

> Nanite = 원본을 잘게 쪼개 저장해 두고, 지금 화면에 필요한 조각만 꺼내 쓰는 방식

---

### Nanite도 map을 구워서 처리할까?
Nanite는 지오메트리 전처리 + 내부 데이터화로 만들어 진다

1. **전통적 의미의 핵심**
    - 고폴리 -> 저폴리 + normal map bake
2. **Nanite의 핵심**
    - 고폴리 -> Nanite 내부 표현(클러스터 계층, 정밀도 설정, fallback mesh 등)

> 즉, Nanite는 지오메트리 자체를 가상화해서 저장/ 스트리밍 /선별하는 기술

```
그럼 어디에 저장되나?

개념적으로는 메시 자산의 Nanite용 빌드 결과로 저장되고, 런타임에 필요한 클러스터가 디스크에서 스트리밍된다.
또한 필요 시 fallback mesh도 함께 생성될 수 있다
```

---

### Nanite는 언리얼 엔진에서 어떻게 처리될까?

| 구분     | 누가 주로 담당?   | 설명      |
| --------- | ----- | ---------- |
| Nanite 데이터 생성  | CPU        | 임포트/빌드 시 내부 표현 생성   |
| 씬 제출/엔진 일반 렌더링 흐름  | 게임 스레드 + 렌더 스레드 + RHI 제출 흐름 | 다른 렌더 기능과 마찬가지로 엔진의 일반 프레임 구조를 탐 |
| 가시성 세분화, 클러스터 컬링, Nanite 래스터 처리 | GPU 중심     | Nanite의 핵심 성능 포인트   |

> Nanite는 CPU가 하던 메시 관리 부담 일부를 GPU 친화적인 방식으로 재편한 렌더링 기술에 가깝다

---

## Nanite의 가상화 지오메트리
### 가상화라고 부르는 이유
- 메시를 전통적인 **고정 LOD 세트로 다루지 않음**
- 내부적으로 더 잘게 쪼개진 **계층 구조와 스트리밍 단위로 관리**하기 때문

> 즉 텍스처 가상화처럼, 지오메트리도 '필요한 것만 꺼내 쓴다'는 발상

- 과정

```
1. 메시를 계층적 클러스터들로 재구성
2. 각 클러스터는 자기 bounds를 가짐
3. 화면에서 안 보이거나 너무 작은 클러스터는 버림
4. 필요한 클러스터만 스트리밍
5. 살아남은 클러스터만 Nanite 경로로 그림
```

---

## 클러스터 컬링?
클러스터 컬링은 메시 전체를 그릴지 말지가 아니라, **메시 내부의 작은 클러스터 묶음을 각각 보이는지 검사해서 버리는 것**

```
클러스터마다
-> 프러스텀 안에 있는가?
-> 다른 물체 뒤에 가려졌는가?
-> 화면에서 차지하는 크기가 충분한가?
-> 지금 시점에 필요한 해상도/디테일인가?

=> 통과한 것만 렌더! 그래서 Nanite는 메시 단위 LOD 교체보다 훨씬 미세한 단위로 디테일을 조절한다
```

---

## Nanite를 활성화하면 가능한 것 vs 불가능한 것
### 가능한 것
- 고밀도 메시를 훨씬 직접적으로 사용
- 수동 LOD/노멀맵 베이킹 부담 감소
- 대규모 디테일 씬을 더 안정적으로 구성
- Virtual Shadow Maps와의 고디테일 섀도우 연동
- 최근 버전에서는 skeletal/foliage/assemblies 방향으로 확장


### 주의 / 제약이 있는 것
- Translucent 기반 워크플로는 제약
- Mesh Decals는 Translucent Blend Mode가 필요하므로 직접 지원되지 않음
- WPO는 가능하지만 비용/제약을 주의해야 함
    - Nanite 메시가 더 작은 클러스터로 분할되어 개별 컬링되므로, 메모리/성능 부담이 커질 수 있음
- Nanite 사용 시 섀도우 쪽은 Movable Lights 또는 Virtual Shadow Maps 쪽을 사실상 같이 봐야 함

> 중요 : Nanite를 켜면 무조건 아무 메시나 공짜로 빨라지는 것은 아니다
