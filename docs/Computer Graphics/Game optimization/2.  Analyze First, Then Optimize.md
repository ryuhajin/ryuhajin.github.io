# 최적화의 황금률
- 분석 먼저, 최적화는 그다음(Analyze first, then optimize)

> 분석 없이 무작정 최적화하면 그래픽 품질만 낮아지고 성능은 전혀 개선되지 않을 수 있다!

---

# 성능 분석: CPU 바운드(CPU‑bound) vs GPU 바운드(GPU‑bound) 판별법
## 1. 왜 원인을 먼저 찾아야 하는가?
- 게임은 매 프레임을 렌더링하기 위해 **CPU**와 **GPU**가 함께 일함
- 두 프로세서는 각자 자신의 작업을 마친 후 상대방이 끝날 때까지 기다림

```
두 프로세서의 작업 속도는 항상 일치하지 않는다
```

1. **동기화 원리**
    - 두 프로세서는 현재 프레임의 작업을 마치면 **상대방이 끝날 때까지 기다림**
2. **병목(Bottleneck)**
    - **더 오래 걸리는 프로세서가 전체 게임의 속도(FPS)를 결정**
3. **최적화의 오류**
    - 이미 빨리 끝나는 프로세서를 더 빠르게 최적화해봤자, 결국 **느린 프로세서를 기다리는 '대기 시간'만 늘어날 뿐 전체 프레임 레이트는 전혀 오르지 않음**

---

## 2. CPU‑bound vs GPU‑bound
장면에서 프레임 속도가 CPU에 의해 제한되는지 아니면 GPU에 의해 제한되는지 확인하기

| 구분 | 의미 | 특징 |
|------|------|------|
| **CPU‑bound** | CPU가 GPU보다 더 오래 걸리는 상태 | CPU 작업(드로우콜 준비, 물리, 게임 로직 등)이 병목 |
| **GPU‑bound** | GPU가 CPU보다 더 오래 걸리는 상태 | GPU 작업(픽셀 셰이더, 폴리곤 처리, 후처리 등)이 병목 |

---

### 간단한 병목 진단 방법: 해상도 절반으로 낮추기
프로파일링 툴이 없더라도 논리적인 테스트를 통해 어느 쪽이 문제인지 판별할 수 있다

1. **독립 실행형 빌드**에서 게임 실행 후 현재 프레임 레이트(FPS) 또는 프레임 타임(ms) 확인
    - 프레임 측정을 통해 기준점 확보
2. **화면 해상도를 원래의 50%로 낮춤**
    - GPU의 픽셀 연산 부하를 강제로 줄임
3. **프레임 변화 관찰**

| 상황 | 진단 | 해야 할 일 |
|------|------|--------------|
| 프레임이 **거의 그대로** (변화 없음) | **CPU‑bound** | CPU 작업(드로우콜, 게임 스레드 등) 최적화 |
| 프레임이 **크게 향상** (예: 60→100 FPS) | **GPU‑bound** | GPU 작업(셰이더, 폴리곤, 텍스처 등) 최적화 |

### 1. 라이트 복잡도 (Light Complexity)
**과도한 광원 중첩 및 큰 감쇠 반경 확인**

- **정상 상태**: 파란색 위주 (가끔 녹색)
- **문제 상태**: 흰색, 보라색 영역
- **해결 방법**
  - 광원 중첩 시 불필요한 라이트 삭제
  - 라이트 details 에서
    - `Cast Shadow` 옵션 비활성화
    - 감쇠 반경(attenuation radius)을 실제 영향 영역으로 축소

```
# 뷰 모드에서 볼 수 있는 최적화 개념 추후 더 배워서 업데이트 하기
```

## stat 명령어 활용하여 디버깅하기
- [게임 통계를 표시하는 데 특화된 콘솔 명령어 매뉴얼](https://dev.epicgames.com/documentation/unreal-engine/stat-commands-in-unreal-engine)

## stat unit 명령
프로젝트의 프레임, 게임, 드로, GPU, RHIT, DynRes 스레드의 퍼포먼스 정보를 표시

|이름|설명|
|---|---|
Frame	| 게임의 프레임 하나를 생성하는 데 걸린 총 시간. 게임 스레드와 드로 스레드는 둘 다 한 프레임이 끝나기 전에 동기화되므로, 프레임 시간은 다음 스레드 중 하나의 시간과 가까운 경우가 많음|
Game	| 프레임 시간이 게임 시간에 가까운 경우 게임 스레드 때문에 게임 퍼포먼스에 병목 구간(부정적인 영향)이 발생하는 것일 수 있음 |
Draw	| 프레임 시간이 드로 시간에 가까운 경우 렌더링 스레드 때문에 게임 퍼포먼스에 병목 구간이 발생하는 것일 수 있음 |
GPU	| GPU 시간은 비디오 카드가 씬을 렌더링할 때 얼마나 오래 걸리는지 측정. GPU 시간은 프레임과 동기화되므로 프레임 시간과 유사할 확률이 높음 |
RHIT	| RHI 스레드 시간은 보통 프레임과 동기화되므로 프레임 시간과 유사할 확률이 높음 |
DynRes	| 지원되고 활성화된 경우 동적 해상도는 1차 스크린 퍼센티지 X 2차 스크린 퍼센티지를 표시|
