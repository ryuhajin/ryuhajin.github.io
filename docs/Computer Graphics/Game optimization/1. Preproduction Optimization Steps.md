# 1. Preproduction Optimization Steps
게임 최적화 기초: 프리 프로덕션 예산 설정과 프로파일링 환경 구축

최적화는 개발 막바지에 하는 작업이 아니다
- **타겟 하드웨어의 한계를 먼저 테스트**하고
- 그 **한계치(예산)를 각 파트에 할당하여 개발 내내 유지**하는 것이 핵심 로직

---

## 1. 프로젝트 초기에 반드시 정해야 할 3가지
프로젝트 초기에는 다음 4단계의 논리적 흐름에 따라 최적화 기준을 세워야 한다

### (1) 타겟 하드웨어 선택

- 콘솔, PC, 모바일 중 어떤 기기에서 실행할 것인지, PC라면 어느 정도의 그래픽 카드, RAM, 최소 CPU 성능을 요구할 것인지 결정
- 요구 사양을 낮게 잡을수록 잠재적인 유저층은 늘어나지만 최적화 작업은 훨씬 까다로워짐
- 반대로 최첨단 그래픽을 타겟으로 하면 고성능 하드웨어가 필요하므로 유저층이 줄어듦
   * 이는 1프레임당 렌더링에 사용할 수 있는 **시간(Frame Time ms)**의 절대적인 상한선을 결정짓는 기준이 됨

### (2) 타겟 프레임 레이트(FPS) 설정

- 빠른 템포의 액션 게임은 60fps 이상이 필수적이지만, 정적인 탐험이나 퍼즐 게임은 30fps로도 충분할 수 있음
- PC 게이머들은 프레임 레이트에 매우 민감하므로 기준을 잘 세워야 함

> 타겟 하드웨어와 프레임 레이트가 정해지면, 게임이 느려졌을 때 이를 판별할 수 있는 명확하고 구체적인 기준점이 생긴다

---

### (3) 타겟 하드웨어 한계치 파악 : 타겟 기기를 정한 뒤 해당 하드웨어의 특성과 한계 테스트

- 목표 프레임을 유지하면서 화면에 그릴 수 있는 폴리곤 수는 몇 개인지, 드로우 콜(Draw Calls)은 몇 번까지 허용되는지 파악

| 테스트 내용 | 방법 |
|-------------|------|
| 최대 폴리곤 수 | 화면을 임시 메시로 가득 채운 후, 프레임이 목표 이하로 떨어질 때까지 개수 증가 |
| 최대 드로우콜 수 | 위와 동일한 방식으로 드로우콜 증가시키며 테스트 |
| 후처리 효과 영향 | 안티에일리어싱, 앰비언트 오클루전, 모션블러, 컬러 그레이딩, 피사계 심도(DOF)를 켰을 때 프레임 변화 측정 |

> 화면에 임시 메시를 수천 개씩 띄워가며 프레임이 떨어지는 한계점을 직접 테스트하여 수치화하기

---

### (4) 자원 할당 : 확보된 전체 자원을 각 콘텐츠 카테고리별로 나누어 '성능 예산(Budget)' 할당

- 확보된 전체 자원(프레임 타임, 메모리, 폴리곤, 드로우 콜 등)을 각 콘텐츠 카테고리별로 나누기
    - 산출된 전체 자원 한계치를 각 개발 파트(환경 아트, 캐릭터, VFX, UI, 포스트 프로세싱 등)에 수치적으로 분할 할당
    - 예: 전체 드로우 콜 한계가 4000이라면, 환경 2000, 캐릭터 1000, VFX 1000 식으로 제한

> 환경 아트는 캐릭터와 VFX, UI를 끈 상태에서 X 밀리초 이내에 렌더링되어야 하며, 드로우 콜은 X 이하, 폴리곤은 X 이하로 맞춰야 한다는 명확한 가이드라인 설정

---

## (5) 성능 측정시 환경 설정 규칙
- Unreal : 프레임/타임 표시 `stat FPS`, `stat unit graph`

### 1. 에디터가 아닌 스탠드얼론 빌드(Standalone Build)에서 측정

  - 엔진 에디터 외부에서 독립적으로 실행되는 빌드 버전을 만들어 테스트
  - 에디터 자체가 소모하는 메모리와 연산(Overhead) 비용이 결과를 심각하게 왜곡

---

### 2. 고정된 카메라 위치 사용

  - 성능을 분석할 때는 **매번 정확히 똑같은 지점과 각도에서 측정**할 수 있도록 '고정된 카메라 위치'나 '고정된 경로'를 사용하기
  - 프레임이 가장 떨어지는 병목 구간들을 찾아 그곳에 고정 카메라를 설치

---

### 3. 측정 변수(동적 요소) 완벽 차단

  - 성능 데이터를 기록하기 전, **실행할 때 마다 결과를 달라지게 만들 수 있는 게임 내의 모든 요소를 끄기**
  - 환경 아트의 성능만 측정한다면 캐릭터, VFX, 포스트 프로세스를 모두 비활성화
  - **무작위로 움직이는 캐릭터나 시시각각 변하는 시간/날씨 시스템은 프레임별 편차를 만든다**
    - 캐릭터는 끄고 **시간과 날씨는 가장 성능을 많이 사용하는 것으로 고정**해놓기 
  - 오직 측정하고자 하는 대상 하나만 남기고 철저히 격리하자
    - **프레임 속도를 인위적으로 제한하는 수직동기화(VSync)나 프레임 스무딩 옵션도 반드시 끄기**

---

### 4. 데이터 추적을 위한 측정 자동화

- 자주 입력하는 디버깅 명령어들을 **하나의 버튼 또는 명령어**로 묶어두기
  - 예: `stat unit graph`, `stat FPS`, 특정 렌더링 설정 해제 등
- 실수로 누락되는 것을 방지하고, 측정 간 일관성을 보장함

---

### 5. 데이터 기록 방법

1. **각 고정 카메라 위치에서 초기 통계 기록하기**
    - 프레임 타임, GPU/CPU 시간, 드로우콜, 폴리곤, 메모리 등
    - 스프레드 시트 등 카메라 위치별로 통계 기록 정리
2. **레벨에 큰 변화가 생길 때 마다 동일 데이터 다시 수집**
    - 상황이 개선되었는지 악화되었는지 확인하기
3. **그래프 등 개선 추이를 시각화하여 팀에 공유하기**

##  최종 체크리스트 (분석 전 확인 사항)

- [ ] 타겟 하드웨어와 프레임 레이트가 결정되었는가?
- [ ] 하드웨어 성능 한계(폴리곤, 드로우콜 등)를 숫자로 알고 있는가?
- [ ] 콘텐츠 유형별 예산이 분배되어 있는가?
- [ ] 상시 성능 지표(On‑Screen)가 팀 전체에 표시되고 있는가?
- [ ] 측정은 **독립 실행형 빌드**에서, **타겟 하드웨어**에서 수행하는가?
- [ ] 카메라 위치/경로, 시간대, 랜덤 요소 등이 모두 고정되었는가?
- [ ] VSync, 프레임 스무딩 등 프레임 제한 요소가 모두 꺼져 있는가?
- [ ] 측정 과정이 자동화(매크로/단일 명령어)되어 일관성을 유지하는가?
- [ ] 성능 데이터가 스프레드시트에 기록되고 그래프로 시각화되고 있는가?
